# 异常处理规范

## 一、异常类层次结构

```
RuntimeException
├── BizException (业务异常基类)
│   ├── UserException        # 用户模块
│   ├── OrderException       # 订单模块
│   ├── PayException         # 支付模块
│   ├── CollectionException  # 藏品模块
│   ├── TradeException       # 交易模块
│   ├── ChainException       # 区块链模块
│   ├── InventoryException   # 库存模块
│   ├── BlindBoxException    # 盲盒模块
│   ├── AuthException        # 认证模块
│   └── AdminException       # 管理模块
│
└── SystemException (系统异常基类)
    └── RemoteCallException  # 远程调用异常
```

**基础异常类位置**：`nft-turbo-common/nft-turbo-base/src/main/java/cn/hollis/nft/turbo/base/exception/`

---

## 二、异常分类与使用场景

| 异常类型 | 使用场景 | 示例 |
|---------|---------|------|
| `BizException` | 业务规则校验失败、业务流程异常 | 库存不足、用户未认证、订单状态不允许 |
| `SystemException` | 系统级错误、基础设施异常 | 限流熔断、远程调用失败、配置错误 |
| `原生异常` | 定时任务、消息监听器等非 HTTP 场景 | 任务执行失败、消息消费失败 |

## 三、不同场景的异常处理规范

### 3.1 Controller / Facade 层（HTTP 请求）

**规范**：抛出自定义业务异常，由全局异常处理器统一处理

```java
// ✅ 正确示例
@PostMapping("/create")
public Result<OrderVO> createOrder(@RequestBody OrderCreateRequest request) {
    // 业务逻辑中抛出自定义异常
    if (inventory < request.getQuantity()) {
        throw new OrderException(OrderErrorCode.INVENTORY_NOT_ENOUGH);
    }
    return Result.success(orderService.create(request));
}

// ❌ 错误示例
@PostMapping("/create")
public Result<OrderVO> createOrder(@RequestBody OrderCreateRequest request) {
    throw new RuntimeException("库存不足");  // 不要抛出原生异常
}
```

**全局异常处理器**：`GlobalWebExceptionHandler` 位于 `nft-turbo-web` 模块

### 3.2 Service 层（业务逻辑）

**规范**：使用 Assert 断言或直接抛出模块对应的异常

```java
// ✅ 方式一：Assert 断言（推荐，代码简洁）
public void updateOrder(Order order) {
    boolean result = orderMapper.updateById(order);
    Assert.isTrue(result, () -> new OrderException(OrderErrorCode.UPDATE_ORDER_FAILED));
}

// ✅ 方式二：直接 throw
public void checkUserStatus(User user) {
    if (user.getState() != UserStateEnum.ACTIVE) {
        throw new UserException(UserErrorCode.USER_STATUS_ABNORMAL);
    }
}

// ❌ 错误示例：在 order 模块抛出 UserException
public void createOrder(OrderRequest request) {
    throw new UserException(UserErrorCode.XXX);  // 应该抛出 OrderException
}
```

### 3.3 定时任务（XXL-Job）

**规范**：记录日志 + 控制任务状态，不需要自定义业务异常

```java
@XxlJob("orderTimeoutJob")
public ReturnT<String> execute() {
    try {
        List<Order> orders = orderService.queryTimeoutOrders();
        orders.forEach(this::processSingle);
        return ReturnT.SUCCESS;
    } catch (Exception e) {
        LOG.error("orderTimeoutJob 执行失败", e);
        return ReturnT.FAIL;  // 或 throw e 让 XXL-Job 感知失败
    }
}

// 单条处理：失败不影响整体
private void processSingle(Order order) {
    try {
        orderService.timeout(order);
    } catch (Exception e) {
        LOG.error("订单超时处理失败, orderId={}", order.getId(), e);
        // 不抛出，继续处理下一条
    }
}
```

### 3.4 消息监听器（MQ Consumer）

**规范**：类似定时任务，日志记录为主，根据业务决定是否重试

```java
@RocketMQMessageListener(topic = "ORDER_TOPIC", consumerGroup = "order-consumer")
public class OrderMessageListener implements RocketMQListener<OrderMessage> {
    
    @Override
    public void onMessage(OrderMessage message) {
        try {
            orderService.process(message);
        } catch (BizException e) {
            // 业务异常：记录日志，不重试
            LOG.error("消息处理业务异常, messageId={}", message.getId(), e);
        } catch (Exception e) {
            // 系统异常：记录日志，抛出让 MQ 重试
            LOG.error("消息处理系统异常, messageId={}", message.getId(), e);
            throw e;
        }
    }
}
```

### 3.5 远程调用（Dubbo/Feign）

**规范**：包装为 SystemException 或 RemoteCallException

```java
public UserInfo getUser(Long userId) {
    try {
        Response<UserInfo> response = userFacadeService.queryById(userId);
        if (!response.getSuccess()) {
            throw new RemoteCallException(RemoteErrorCode.REMOTE_CALL_FAILED);
        }
        return response.getData();
    } catch (Exception e) {
        LOG.error("远程调用用户服务失败, userId={}", userId, e);
        throw new SystemException(SystemErrorCode.REMOTE_SERVICE_ERROR);
    }
}
```

## 四、错误码规范

### 4.1 错误码接口定义

```java
public interface ErrorCode {
    String getCode();
    String getMessage();
}
```

### 4.2 模块错误码枚举示例

```java
public enum OrderErrorCode implements ErrorCode {
    
    // 格式：错误类型_具体描述
    ORDER_NOT_FOUND("ORDER_NOT_FOUND", "订单不存在"),
    ORDER_STATUS_INVALID("ORDER_STATUS_INVALID", "订单状态不允许此操作"),
    INVENTORY_NOT_ENOUGH("INVENTORY_NOT_ENOUGH", "库存不足"),
    CREATE_ORDER_FAILED("CREATE_ORDER_FAILED", "创建订单失败"),
    UPDATE_ORDER_FAILED("UPDATE_ORDER_FAILED", "更新订单失败");
    
    private final String code;
    private final String message;
    
    OrderErrorCode(String code, String message) {
        this.code = code;
        this.message = message;
    }

    @Override
    public String getCode() {
        return code;
    }

    @Override
    public String getMessage() {
        return message;
    }
}
```

## 五、异常处理速查表

| 代码位置 | 异常类型 | 处理方式 | 示例 |
|---------|---------|---------|------|
| Controller/Facade | `XxxException` | 抛出，全局处理器捕获 | `throw new OrderException(...)` |
| Service 业务校验 | `XxxException` | Assert 或直接 throw | `Assert.isTrue(...)` |
| Service 数据操作 | `BizException` | Assert + RepoErrorCode | `Assert.isTrue(result, ...)` |
| 定时任务-整体 | 原生异常 | try-catch + 日志 + ReturnT | `return ReturnT.FAIL` |
| 定时任务-单条 | 原生异常 | try-catch + 日志，不抛出 | `LOG.error(...)` |
| 消息监听器 | 原生异常 | 业务异常不重试，系统异常重试 | 见 3.4 示例 |
| 远程调用 | `SystemException` | 包装后抛出 | `throw new SystemException(...)` |

## 六、最佳实践

### 6.1 DO ✅

- 使用模块对应的异常类（OrderException、UserException 等）
- 使用 Assert 断言简化代码
- 错误码要有明确的业务含义
- 异常信息要包含关键业务参数（便于排查）
- 定时任务/消息监听器要有完整的日志记录

### 6.2 DON'T ❌

- 不要在业务代码中抛出 RuntimeException、Exception 等原生异常
- 不要在 A 模块抛出 B 模块的异常
- 不要吞掉异常不处理（至少要记录日志）
- 不要在异常信息中暴露敏感信息（密码、密钥等）
- 不要用异常控制业务流程（异常应该是异常情况）

## 八、全局异常处理器说明

全局异常处理器位于 `nft-turbo-web` 模块：

`cn.hollis.nft.turbo.web.handler.GlobalWebExceptionHandler`

处理逻辑：

| 异常类型 | HTTP 状态码 | 返回内容 |
|---------|------------|---------|
| `MethodArgumentNotValidException` | 400 | 字段校验错误详情 |
| `BizException` | 200 | 业务错误码 + 错误信息 |
| `SystemException` | 200 | 系统错误码 + 错误信息 |
| `Throwable` | 200 | 通用错误提示："哎呀，当前网络比较拥挤，请您稍后再试~" |

## 九、常见问题

### Q1: 什么时候用 BizException，什么时候用具体的 XxxException？

**A**: 优先使用具体的模块异常类（如 OrderException），只有在通用场景（如数据库操作失败）才使用 BizException + RepoErrorCode。

### Q2: 定时任务中调用的 Service 方法抛出了 BizException 怎么办？

**A**: 在定时任务中 catch 住，记录日志，根据业务决定是否继续处理下一条。

### Q3: 如何在异常信息中包含业务参数？

**A**: 使用带 message 参数的构造函数：
```java
throw new OrderException("订单不存在, orderId=" + orderId, OrderErrorCode.ORDER_NOT_FOUND);
```

### Q4: Assert 断言用的是哪个包？

**A**: 使用 Hutool 的 Assert：`cn.hutool.core.lang.Assert`
